The cold path

Somewhere right now, a database server is billing you to do nothing.

The standard stack asks you to pay three taxes before your first user shows up: a machine that never sleeps, a shard key you must guess before you have traffic, and an ops rotation for the one component that can't go down. None of these are laws of nature. They're habits.

Tax nº1 — the idle server

Always on, mostly waiting

Your database bills by the hour whether or not anyone writes. Scale-to-zero exists for compute — but state stayed behind, chained to a daemon.

Tax nº2 — the shard key

Guessed early, paid forever

Horizontal scale asks you to predict your access patterns before you have any. Guess wrong and the fix is called a migration, and nobody schedules those for fun.

Tax nº3 — the cold start

Restart means rebuild

Caches empty. Placement forgotten. Connections re-herded. Everything your system learned about itself, discarded on every deploy.

How it stays warm

Three mechanics. No magic, only physics.

Every object is a real SQL database

Not a JSON blob with pretensions — an actual SQLite database per object, with tables, indexes, and CHECK constraints. Your invariants live in the schema, where they can't be forgotten by the one code path someone wrote at 2am.

Transactions declare their participants up front and commit atomically across all of them. A constraint violation in one object rolls back every object. The transfer-between-accounts problem is an API call, not an architecture meeting.

# an "object" is one entity with real schema POST /objects/alice/exec { "ops": [ { "sql": "CREATE TABLE account ( balance INTEGER NOT NULL CHECK (balance >= 0))" }, { "sql": "INSERT INTO account VALUES (?1)", "params": [100] } ]}

The commit point is one write

Object storage is the only durable truth. Local disks hold disposable working copies — which is why ephemeral containers, spot instances, and kill -9 are all fine. A transaction stages its snapshots, then commits with a single blob write. Crash before it: nothing happened. Crash after it: recovery rolls forward at the next boot, mechanically.

No quorums to keep fed. No replication topology to tend. The consensus you need is rented from the storage layer's conditional writes, and you only pay for it when ownership moves.

# the entire commit protocol 1. run SQL in local txns (undoable) 2. snapshot + stage to bucket (undoable) 3. PUT txns/<id>.json ← COMMIT 4. promote snapshots (roll-forward) # crash anywhere: boot recovery finishes # the sentence. no operator required.

Placement anneals itself

Objects that transact together migrate together — automatically, because moving an object is a routing edit, not a data copy. One-off transactions don't displace settled data (it snaps back home), but a repeated partnership earns a move-in. Load pressure pushes back so no node becomes the hot spot.

The result is the simulation at the top of this page: coordination overhead falls from random-placement chaos to the workload's floor, while you write exactly zero placement configuration. And the learning is durable — restart the whole fleet and it resumes where it left off.

# measured, not promised window cross-node migrations 0..1000 30.4% 304 3000..4000 14.2% 142 9000..10000 13.3% 134 -- stop the world, restart -- 0..1000 15.5% 155 # the fleet remembered. floor ≈ noise rate.
Program against it

An API you already know how to use.

HTTP and JSON. SQL you've written a thousand times. Clients are one file with zero dependencies — vendoring is copying.

# atomic transfer across two objects — participants declared up front curl -s localhost:8787/txn -d '{ "objects": ["alice", "bob"], "ops": [ {"object": "alice", "sql": "UPDATE account SET balance = balance - ?1", "params": [60]}, {"object": "bob", "sql": "UPDATE account SET balance = balance + ?1", "params": [60]} ]}' # {"txn_id":"w17-4","results":[{"rows_affected":1},{"rows_affected":1}]} # if alice would overdraft, bob's credit rolls back too. always.
# clients/scalewarm.py — stdlib only, copy it into your repo db = ScaleWarm() db.txn(["alice", "bob"], [ ("alice", "UPDATE account SET balance = balance - ?1", [60]), ("bob", "UPDATE account SET balance = balance + ?1", [60]), ]) rows = db.query("alice", "SELECT balance FROM account") # [{"balance": 40}]
// clients/scalewarm.ts — fetch only: Node, Bun, Deno, browsers, Workers const db = new ScaleWarm(); await db.txn(["alice", "bob"], [ { object: "alice", sql: "UPDATE account SET balance = balance - ?1", params: [60] }, { object: "bob", sql: "UPDATE account SET balance = balance + ?1", params: [60] }, ]); const rows = await db.query("alice", "SELECT balance FROM account"); // [{ balance: 40 }]

Fine print, in large type

When you should not use ScaleWarm.

  • Big objects. Durability is snapshot-per-commit: a 100 MB object pays 100 MB per write transaction. Keep objects entity-sized. (Delta shipping is on the roadmap; today this is the rule.)
  • Fleet-wide analytics. There is no global SQL engine. "Average balance across all accounts" means fanning out over objects. Ship changes to a warehouse for that.
  • One supernova object. Each object has exactly one writer at a time — that's where the guarantees come from. Optimistic batching takes a single hot object to tens of thousands of writes per second; past one core's serial ceiling, you need redesign, not more nodes.
  • Sub-millisecond commits. The commit point is a blob write. On local disk that's fast; against remote object storage it's a network round trip. Physics is undefeated.
  • Decades of scar tissue. Mature databases have survived failure modes we haven't met yet. If your blast radius is measured in careers, bring the grown-ups.

Why tell you this? Because every system that hid its tradeoffs eventually introduced you to them personally, at 3am. Our pitch is that the guarantees are itemized — including the ones we don't sell.
